**IMPORTANT pino convention:** The first argument to `log.error()` is always the data object `{ err: error }` (using `err` key, NOT `error`). The second argument is the message string. This is the opposite of console.error's argument order. Using `err` as the key triggers pino's built-in error serializer which extracts stack trace, message, and code.

**Add Sentry capture** in every catch block, right after the `log.error` call:

```typescript
Sentry.withScope((scope) => {
  scope.setTag("correlation_id", correlationId);
  // Add org_id and session_id tags when available in that route's scope
  if (orgId) scope.setTag("org_id", orgId);
  if (sessionId) scope.setTag("session_id", sessionId);
  Sentry.captureException(error);
});
```

Which tags are available varies by route:
- **webhooks/stripe**: `event.id` available, org_id from subscription lookup if present
- **webhooks/payment**: `orgId` from header resolution
- **webhooks/twilio**: `sms.from` available, no org_id directly
- **webhooks/vapi**: no context available (set correlation_id only)
- **widget/response**: `sessionId` from request body
- **widget/session**: `sessionId` from URL params
- **signup/checkout**: `email` from body
- **billing/checkout**: `user.id` available
- **billing/portal**: `user.id` available
- **cron/outreach**: no context (set correlation_id only)
- **cron/tasks**: no context (set correlation_id only)
- **org/create**: `user.id` available

Set whatever context variables are available in that route's scope. Don't force-add variables that aren't in scope.
